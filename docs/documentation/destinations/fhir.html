FHIR Destination

Send FHIR resources to a FHIR-compliant server with native support for FHIR operations.

Overview

The FHIR destination provides a purpose-built integration with FHIR (Fast Healthcare Interoperability Resources) servers. Unlike the generic HTTP destination, the FHIR destination understands FHIR-specific operations (create, update, transaction bundles), handles resource versioning, and validates FHIR version compatibility. It automatically constructs the correct REST endpoints based on resource types and supports both individual resource operations and FHIR transaction bundles.

Configuration

Define a FHIR destination in your root intu.yaml under the destinations key:

yaml
# intu.yaml
destinations:
  ehr-fhir-server:
    type: fhir
    fhir:
      base_url: https://fhir.hospital.org/fhir
      version: R4
      operations:
        - create
        - update
        - transaction
      auth:
        type: oauth2
        token_url: https://auth.hospital.org/oauth/token
        client_id: ${FHIR_CLIENT_ID}
        client_secret: ${FHIR_CLIENT_SECRET}
        scopes:
          - system/Patient.write
          - system/Encounter.write
          - system/Observation.write
      tls:
        ca_cert: /etc/ssl/certs/hospital-ca.pem
      timeout_ms: 30000

Properties

base_url string required
Base URL of the FHIR server. Resource type and ID are appended automatically based on the operation. Supports environment variable interpolation.
version string optional
FHIR version. One of: R4, STU3. Affects content type headers and operation behavior. Defaults to R4.
operations string[] optional
Allowed FHIR operations. Values: create (POST), update (PUT), transaction (Bundle POST to base URL). Defaults to ["create"]. The transformer determines which operation to use per message.
timeout_ms int optional
Request timeout in milliseconds. Defaults to 30000.
validate_resources bool optional
Validate outbound resources against the FHIR schema before sending. Defaults to false.

Authentication

The auth object supports the same authentication types as the HTTP destination: basic, bearer, api_key, and oauth2.

yaml
# OAuth2 (recommended for FHIR SMART on FHIR backends)
auth:
  type: oauth2
  token_url: https://auth.hospital.org/oauth/token
  client_id: ${FHIR_CLIENT_ID}
  client_secret: ${FHIR_CLIENT_SECRET}
  scopes:
    - system/Patient.write
    - system/Observation.write

# Bearer token (for simpler setups)
auth:
  type: bearer
  token: ${FHIR_ACCESS_TOKEN}

TLS

tls.ca_cert string optional
Path to a PEM-encoded CA certificate.
tls.client_cert string optional
Path to a client certificate for mutual TLS.
tls.client_key string optional
Path to the client private key for mutual TLS.
tls.insecure_skip_verify bool optional
Skip server certificate verification. Defaults to false.

Full Example

Receive HL7v2 ADT and ORU messages, transform them into FHIR Patient, Encounter, and Observation resources, and push them to a FHIR R4 server as a transaction bundle.

Root Configuration

yaml
# intu.yaml
destinations:
  clinical-fhir:
    type: fhir
    fhir:
      base_url: https://fhir.hospital.org/fhir
      version: R4
      operations:
        - create
        - update
        - transaction
      validate_resources: true
      auth:
        type: oauth2
        token_url: https://auth.hospital.org/oauth/token
        client_id: ${FHIR_CLIENT_ID}
        client_secret: ${FHIR_CLIENT_SECRET}
        scopes:
          - system/Patient.write
          - system/Encounter.write
          - system/Observation.write
      tls:
        ca_cert: /etc/ssl/certs/hospital-ca.pem
      timeout_ms: 45000

Channel Configuration

yaml
# channels/hl7-to-fhir/channel.yaml
name: hl7-to-fhir
description: Transform HL7v2 messages to FHIR transaction bundles

source:
  type: tcp
  tcp:
    port: 6670
    mode: mllp

destinations:
  - clinical-fhir

Destination Transformer

typescript
// channels/hl7-to-fhir/transformer.ts
import { Message, Context } from "@intu/sdk";

export default function transform(msg: Message, ctx: Context): Message {
  const hl7 = msg.body;
  const pid = hl7.segments?.find((s: any) => s.name === "PID");
  const pv1 = hl7.segments?.find((s: any) => s.name === "PV1");
  const obxSegments = hl7.segments?.filter((s: any) => s.name === "OBX") ?? [];

  const patientId = pid?.fields[3]?.components[0] ?? "";

  const patient = {
    resourceType: "Patient",
    identifier: [
      {
        system: "urn:oid:2.16.840.1.113883.4.1",
        value: patientId,
      },
    ],
    name: [
      {
        family: pid?.fields[5]?.components[0] ?? "",
        given: [pid?.fields[5]?.components[1] ?? ""],
      },
    ],
    gender: mapGender(pid?.fields[8]?.components[0]),
    birthDate: formatFhirDate(pid?.fields[7]?.components[0]),
  };

  const entries: any[] = [
    {
      resource: patient,
      request: {
        method: "PUT",
        url: `Patient?identifier=urn:oid:2.16.840.1.113883.4.1|${patientId}`,
      },
    },
  ];

  if (pv1) {
    const encounter = {
      resourceType: "Encounter",
      status: "in-progress",
      class: {
        system: "http://terminology.hl7.org/CodeSystem/v3-ActCode",
        code: mapEncounterClass(pv1.fields[2]?.components[0]),
      },
      subject: {
        reference: `Patient?identifier=urn:oid:2.16.840.1.113883.4.1|${patientId}`,
      },
      period: {
        start: formatFhirDateTime(pv1.fields[44]?.components[0]),
      },
    };
    entries.push({
      resource: encounter,
      request: { method: "POST", url: "Encounter" },
    });
  }

  for (const obx of obxSegments) {
    const observation = {
      resourceType: "Observation",
      status: "final",
      code: {
        coding: [
          {
            system: "http://loinc.org",
            code: obx.fields[3]?.components[0] ?? "",
            display: obx.fields[3]?.components[1] ?? "",
          },
        ],
      },
      subject: {
        reference: `Patient?identifier=urn:oid:2.16.840.1.113883.4.1|${patientId}`,
      },
      valueQuantity: {
        value: parseFloat(obx.fields[5]?.components[0] ?? "0"),
        unit: obx.fields[6]?.components[0] ?? "",
      },
      referenceRange: [
        { text: obx.fields[7]?.components[0] ?? "" },
      ],
    };
    entries.push({
      resource: observation,
      request: { method: "POST", url: "Observation" },
    });
  }

  const bundle = {
    resourceType: "Bundle",
    type: "transaction",
    entry: entries,
  };

  ctx.logger.info(
    `Created FHIR transaction bundle with ${entries.length} entries for patient ${patientId}`
  );

  return {
    ...msg,
    body: bundle,
    headers: {
      ...msg.headers,
      "fhir-operation": "transaction",
    },
  };
}

function mapGender(code: string | undefined): string {
  switch (code) {
    case "M": return "male";
    case "F": return "female";
    case "O": return "other";
    default:  return "unknown";
  }
}

function mapEncounterClass(code: string | undefined): string {
  switch (code) {
    case "I": return "IMP";
    case "O": return "AMB";
    case "E": return "EMER";
    default:  return "AMB";
  }
}

function formatFhirDate(raw: string | undefined): string | undefined {
  if (!raw || raw.length < 8) return undefined;
  return `${raw.slice(0, 4)}-${raw.slice(4, 6)}-${raw.slice(6, 8)}`;
}

function formatFhirDateTime(raw: string | undefined): string | undefined {
  if (!raw || raw.length < 8) return undefined;
  const date = `${raw.slice(0, 4)}-${raw.slice(4, 6)}-${raw.slice(6, 8)}`;
  if (raw.length >= 14) {
    return `${date}T${raw.slice(8, 10)}:${raw.slice(10, 12)}:${raw.slice(12, 14)}Z`;
  }
  return date;
}
FHIR vs HTTP Destination Use the FHIR destination instead of the generic HTTP destination when you need FHIR-aware behavior: automatic endpoint construction from resource types, FHIR version-specific content types, transaction bundle support, and optional resource validation. For simple POST-to-URL scenarios, the HTTP destination may be simpler.
Tip Use transaction bundles to send multiple related resources (Patient, Encounter, Observations) atomically. The FHIR server processes the entire bundle as a single unit, ensuring referential integrity.

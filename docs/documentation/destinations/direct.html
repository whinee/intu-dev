Direct Destination

Send clinical documents via Direct messaging (Direct Project) for secure healthcare communication.

Overview

The Direct destination sends messages using the Direct protocol (also known as Direct Project or Direct Secure Messaging), a standards-based method for secure, point-to-point healthcare communication. Direct messaging is mandated by the ONC for health information exchange in the United States and is used to send clinical summaries (C-CDAs), referral documents, lab results, and discharge summaries between healthcare providers. Messages are encrypted using S/MIME and delivered over SMTP to Direct addresses.

Configuration

Define a Direct destination in your root intu.yaml under the destinations key:

yaml
# intu.yaml
destinations:
  referral-direct:
    type: direct
    direct:
      to: dr.smith@direct.partnerhospital.org
      from: intu-referrals@direct.hospital.org
      certificate: /etc/ssl/certs/direct-signing.pem
      smtp_host: direct-smtp.hospital.org
      smtp_port: 465
      tls:
        enabled: true

Properties

to string required
The recipient's Direct address (e.g. provider@direct.hospital.org). Direct addresses follow the same format as email but operate on a separate trust infrastructure.
from string required
The sender's Direct address. Must be a registered address on your Direct HISP (Health Information Service Provider).
certificate string optional
Path to the PEM-encoded signing certificate for S/MIME encryption. If omitted, the HISP handles signing.
private_key string optional
Path to the private key corresponding to the signing certificate.
smtp_host string required
Hostname of the Direct HISP's SMTP gateway.
smtp_port int optional
SMTP port. Defaults to 465 (implicit TLS). May also use 587 (STARTTLS).
content_type string optional
MIME type for the message attachment. Defaults to application/xml for CDA documents. Other options: application/pdf, text/plain.

TLS

tls.enabled bool optional
Enable TLS for the SMTP connection. Defaults to true. Direct messaging requires TLS.
tls.ca_cert string optional
Path to a PEM-encoded CA certificate for verifying the HISP's SMTP server.

Full Example

Receive referral orders via HTTP, generate a C-CDA clinical summary, and send it via Direct messaging to the receiving provider.

Root Configuration

yaml
# intu.yaml
destinations:
  specialist-referral:
    type: direct
    direct:
      to: cardiology@direct.heartcenter.org
      from: referrals@direct.hospital.org
      certificate: /etc/ssl/certs/direct-signing.pem
      private_key: /etc/ssl/private/direct-signing-key.pem
      smtp_host: direct-smtp.hospital.org
      smtp_port: 465
      content_type: application/xml
      tls:
        enabled: true
        ca_cert: /etc/ssl/certs/hisp-ca.pem

Channel Configuration

yaml
# src/channels/referral-direct/channel.yaml
name: referral-direct
description: Send referral C-CDA documents via Direct messaging

source:
  type: http
  http:
    port: 8083
    path: /referrals/submit

destinations:
  - specialist-referral

Trust Framework Direct messaging requires that both sender and receiver are registered with a Health Information Service Provider (HISP) and that their certificates are part of the DirectTrust network or an equivalent trust bundle. Messages sent to addresses outside the trust framework will be rejected.
Note Direct messages are encrypted end-to-end using S/MIME. The signing certificate (certificate) is used to digitally sign outbound messages, proving the sender's identity. The recipient's public certificate is resolved automatically through DNS or LDAP by the HISP.

Pipeline & Runtime

Error handling, retry, batch processing, hot reload, storage modes, and tracing

Pipeline Stages

Every message flows through a defined sequence of stages. Understanding the order helps you decide where to place logic and how to configure error handling.

Order Stage Script Description
1Receive—Message arrives from the source connector
2Preprocessorpreprocess.tsModify raw payload before parsing
3Parse—Automatic parsing based on data_types.inbound
4Validatorvalidator.tsAccept or reject the message
5Source Filterfilter.tsDrop messages before transformation (return true/false)
6Transformertransformer.tsMain transformation logic
7Dest Filterdest-filter.tsPer-destination filtering
8Dest Transformerdest-transform.tsPer-destination transformation
9Send—Deliver to each destination connector
10Response Transformerresponse-transform.tsTransform destination response before returning to source
11Postprocessorpostprocess.tsRuns after all destinations have been processed

Pipeline YAML

yaml
pipeline:
  preprocessor: preprocess.ts
  validator: validator.ts
  source_filter: filter.ts
  transformer: transformer.ts
  postprocessor: postprocess.ts

destinations:
  - name: primary-output
    ref: kafka-output
    filter: dest-filter.ts
    transformer:
      entrypoint: dest-transform.ts
    response_transformer:
      entrypoint: response-transform.ts

Error Handling

Error handling is configured per-channel in channel.yaml. It controls what happens when a message fails at any pipeline stage or destination delivery.

on_error Modes

Mode Behavior
continue Log the error and continue processing subsequent messages. The failed message is marked as ERROR in storage.
stop Stop the channel after the error. No further messages are processed until the channel is restarted.
retry Retry the message according to the retry policy before giving up.

Dead Letter Queue (DLQ)

Messages that exhaust all retries or are rejected by the validator can be forwarded to a dead-letter queue for later inspection and reprocessing.

yaml
error_handling:
  on_error: continue
  dlq:
    destination: kafka-dlq
    include_original: true
  alert:
    destination: alert-http

The DLQ message payload includes:

  • OriginalMessageID — ID of the failed message
  • CorrelationID — correlation ID for tracing
  • ChannelID — channel where the error occurred
  • Destination — destination name (if applicable)
  • Error — error message string
  • Timestamp — when the error occurred
  • OriginalPayload — original message content (if include_original: true)

Retry Configuration

Retry policies are configured per-destination and control how failed deliveries are retried.

yaml
destinations:
  - name: primary-output
    ref: http-endpoint
    retry:
      max_attempts: 5
      backoff: exponential
      initial_delay_ms: 100
      max_delay_ms: 30000
      jitter: true
      retry_on:
        - status_5xx
        - timeout
        - connection_refused
      no_retry_on:
        - status_4xx

Backoff Strategies

Strategy Delay Pattern
linear Linearly increasing delay: initial_delay_ms × attempt number, capped at max_delay_ms
exponential Doubling delay: initial_delay_ms × 2(attempt-1), capped at max_delay_ms

Retry Conditions

Condition Description
status_5xxHTTP 500-599 responses
status_4xxHTTP 400-499 responses
timeoutConnection or request timeout
connection_refusedTCP connection refused

Destination Queues

Each destination can have an in-memory queue to buffer messages when the destination is slow or temporarily unavailable.

yaml
destinations:
  - name: primary-output
    ref: http-endpoint
    queue:
      enabled: true
      max_size: 1000
      overflow: drop_oldest    # drop_oldest | reject | block
      threads: 4
Overflow Behavior
drop_oldestEvict the oldest message in the queue to make room
rejectReject new messages when the queue is full
blockBlock the pipeline until space is available

Batch Processing

Batch mode splits a single inbound payload into multiple individual messages, each processed independently through the pipeline.

yaml
batch:
  enabled: true
  type: split
  split_on: newline           # newline | hl7_batch | fhir_bundle | xml_root
  max_batch_size: 1000
  batch_timeout_ms: 5000

Split Types

split_on Description
newlineSplit on newline characters. Each line becomes a separate message.
hl7_batchSplit HL7 batch messages (BHS/BTS wrapped). Each MSH segment starts a new message.
fhir_bundleSplit FHIR Bundle entries. Each Bundle.entry becomes a separate message.
xml_rootSplit on XML root elements. Each top-level element becomes a separate message.

Hot Reload

When running intu serve, the engine watches your project directory for changes using filesystem notifications. Editing any channel.yaml or .ts file triggers an automatic reload of the affected channel.

What triggers a reload

  • Edit channel.yaml — channel config is reloaded and the channel restarts
  • Edit .ts files — TypeScript is recompiled (npm run build), then the channel restarts
  • Add a new channel directory — new channel with channel.yaml is auto-discovered and started
  • Remove a channel directory — the channel is stopped gracefully

Debounce

Rapid file changes are debounced with a 2-second window. If multiple files change within 2 seconds, only one reload is triggered. This prevents unnecessary restarts during save-all operations or when an editor writes multiple files.

Note Hot reload recompiles TypeScript by running npm run build under the hood. If the build fails (syntax errors), the existing channel continues running with the previous code. Check the engine logs for compilation errors.

Correlation ID & Tracing

Configure correlation ID propagation to trace messages across channels, services, and systems.

yaml
tracing:
  correlation_id_header: X-Correlation-ID
  propagate: true

When correlation_id_header is set, intu extracts the correlation ID from the specified HTTP header (or generates one if absent). The ID is propagated through all pipeline stages and included in message metadata, logs, and outbound requests.

Auto-Response

Auto-response automatically generates a response back to the source after processing, without requiring a custom response transformer.

yaml
auto_response:
  enabled: true
  type: hl7_ack
  success_code: AA
  error_code: AE
  reject_code: AR

This is commonly used with TCP/MLLP sources where HL7v2 ACK messages must be returned to the sending system. The ACK message is generated with the correct MSH and MSA segments based on the processing result.

Channel Ordering & Dependencies

Control channel startup order using depends_on and startup_order.

yaml
# channel-b/channel.yaml
id: channel-b
enabled: true
depends_on:
  - channel-a
startup_order: 10

depends_on ensures listed channels start before this one. startup_order provides fine-grained ordering (lower numbers start first).

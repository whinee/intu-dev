Deployment

PostgreSQL, Kafka, Redis, Docker, TLS, authentication, and observability

Overview

In development, intu runs standalone with in-memory storage and no external dependencies. For production, you can enable PostgreSQL (message storage), Redis (cluster coordination), Kafka (message bus), TLS, external authentication, and observability.

Docker Compose

Scaffolded projects include a docker-compose.yml that sets up PostgreSQL and Redis alongside the intu application. This is the fastest way to test production-like configurations locally.

bash
# Start all services (postgres, redis, intu)
docker-compose up --build

# Services available:
#   PostgreSQL: localhost:5432 (user: postgres, password: postgres, db: intu)
#   Redis:      localhost:6379
#   intu:       localhost:8081 (HTTP source), localhost:3000 (dashboard)

Docker Build Only

bash
# Build the Docker image
docker build -t intu-app .

# Run standalone
docker run -p 8081:8081 -p 8082:8082 -p 3000:3000 intu-app

PostgreSQL

PostgreSQL provides persistent, queryable message storage with indexed columns for fast dashboard queries.

Setup

bash
# Start PostgreSQL with Docker
docker run -d --name intu-postgres \
  -e POSTGRES_USER=postgres \
  -e POSTGRES_PASSWORD=postgres \
  -e POSTGRES_DB=intu \
  -p 5432:5432 \
  postgres:16-alpine

# Verify it's running
docker logs intu-postgres

Configuration

yaml
# intu.yaml (or intu.prod.yaml)
message_storage:
  driver: postgres
  mode: full
  postgres:
    dsn: postgres://postgres:postgres@localhost:5432/intu?sslmode=disable
    table_prefix: intu_
    max_open_conns: 25
    max_idle_conns: 5

Using environment variables

yaml
# intu.yaml
message_storage:
  driver: postgres
  mode: full
  postgres:
    dsn: ${INTU_POSTGRES_DSN}
text
# .env
INTU_POSTGRES_DSN=postgres://postgres:postgres@localhost:5432/intu?sslmode=disable

Verify

bash
# Start the engine with postgres profile
intu serve --dir . --profile prod

# Send a test message
curl -X POST http://localhost:8081/ingest \
  -H "Content-Type: application/json" \
  -d '{"test": "postgres"}'

# Verify via CLI
intu message list --dir . --channel http-to-file

# Verify via dashboard API
curl -u admin:admin http://localhost:3000/api/storage-info | jq .

# Verify directly in the database
docker exec -it intu-postgres psql -U postgres -d intu \
  -c "SELECT id, channel_id, stage, status FROM intu_messages LIMIT 5;"

Kafka

Kafka can be used as both a message source (consumer) and destination (producer).

Setup

bash
# Start Kafka with Docker (using KRaft mode, no ZooKeeper)
docker run -d --name intu-kafka \
  -p 9092:9092 \
  -e KAFKA_CFG_NODE_ID=0 \
  -e KAFKA_CFG_PROCESS_ROLES=controller,broker \
  -e KAFKA_CFG_CONTROLLER_QUORUM_VOTERS=0@localhost:9093 \
  -e KAFKA_CFG_LISTENERS=PLAINTEXT://:9092,CONTROLLER://:9093 \
  -e KAFKA_CFG_ADVERTISED_LISTENERS=PLAINTEXT://localhost:9092 \
  -e KAFKA_CFG_CONTROLLER_LISTENER_NAMES=CONTROLLER \
  -e KAFKA_CFG_LISTENER_SECURITY_PROTOCOL_MAP=CONTROLLER:PLAINTEXT,PLAINTEXT:PLAINTEXT \
  bitnami/kafka:latest

# Create topics
docker exec intu-kafka kafka-topics.sh --create --topic intu-input --bootstrap-server localhost:9092
docker exec intu-kafka kafka-topics.sh --create --topic intu-output --bootstrap-server localhost:9092

Kafka Source Configuration

yaml
# channel.yaml - consume from Kafka topic
listener:
  type: kafka
  kafka:
    brokers:
      - localhost:9092
    topic: intu-input
    group_id: intu-consumer
    offset: earliest

Kafka Destination Configuration

yaml
# Named destination in intu.yaml
destinations:
  kafka-output:
    type: kafka
    kafka:
      brokers:
        - localhost:9092
      topic: intu-output

Test Kafka End-to-End

bash
# Start a consumer to watch the output topic
docker exec intu-kafka kafka-console-consumer.sh \
  --bootstrap-server localhost:9092 \
  --topic intu-output \
  --from-beginning &

# Produce a message to the input topic
echo '{"patientId": "P001", "name": "John"}' | \
  docker exec -i intu-kafka kafka-console-producer.sh \
    --bootstrap-server localhost:9092 \
    --topic intu-input

# The transformed message should appear in the consumer output

Redis Cluster

Redis enables multi-instance coordination, channel distribution, and deduplication in clustered deployments.

Setup

bash
# Start Redis with Docker
docker run -d --name intu-redis \
  -p 6379:6379 \
  redis:7-alpine

Configuration

yaml
# intu.prod.yaml
cluster:
  enabled: true
  mode: active-passive
  coordination:
    type: redis
    redis:
      address: localhost:6379
      password: ${INTU_REDIS_PASSWORD}
  deduplication:
    enabled: true
    window: 5m

Test Cluster Coordination

bash
# Start two engine instances (different ports for dashboard)
intu serve --dir . --profile prod &
# Modify dashboard port for second instance
INTU_DASHBOARD_PORT=3001 intu serve --dir . --profile prod &

# Channels should be distributed across instances
# Check logs for "acquired channel" and "released channel" messages

Test Deduplication

bash
# Send the same message ID twice
curl -X POST http://localhost:8081/ingest \
  -H "Content-Type: application/json" \
  -H "X-Message-ID: dedup-test-001" \
  -d '{"test": "first"}'

curl -X POST http://localhost:8081/ingest \
  -H "Content-Type: application/json" \
  -H "X-Message-ID: dedup-test-001" \
  -d '{"test": "duplicate"}'

# Only one message should be processed

Profile-Based Configuration

Use profiles to switch between development and production configurations without modifying the base intu.yaml.

yaml
# intu.dev.yaml
runtime:
  profile: dev
  log_level: debug
  mode: standalone

message_storage:
  driver: memory
  mode: full
yaml
# intu.prod.yaml
runtime:
  profile: prod
  log_level: warn

message_storage:
  driver: postgres
  mode: full
  postgres:
    dsn: ${INTU_POSTGRES_DSN}

cluster:
  enabled: true
  mode: active-passive
  coordination:
    type: redis
    redis:
      address: ${INTU_REDIS_ADDRESS}
bash
# Switch profiles
intu serve --dir . --profile dev    # memory storage, debug logs
intu serve --dir . --profile prod   # postgres storage, redis cluster

TLS Configuration

Enable TLS on HTTP and TCP listeners to encrypt data in transit.

Generate self-signed certificates (for testing)

bash
# Generate a self-signed certificate
openssl req -x509 -newkey rsa:4096 -keyout server.key -out server.crt \
  -days 365 -nodes -subj '/CN=localhost'

HTTP listener with TLS

yaml
listener:
  type: http
  http:
    port: 8443
    path: /ingest
    tls:
      cert_file: ./certs/server.crt
      key_file: ./certs/server.key
bash
# Test with curl (use -k for self-signed certs)
curl -k -X POST https://localhost:8443/ingest \
  -H "Content-Type: application/json" \
  -d '{"test": "tls"}'

Dashboard Authentication

OIDC Setup

Configure OpenID Connect for SSO with providers like Google, Okta, Auth0, or Keycloak.

yaml
dashboard:
  auth:
    type: oidc
    issuer: https://accounts.google.com
    client_id: ${OIDC_CLIENT_ID}
    client_secret: ${OIDC_CLIENT_SECRET}
    redirect_url: http://localhost:3000/auth/callback

LDAP Setup

Configure LDAP for directory-based authentication with Active Directory or OpenLDAP.

yaml
dashboard:
  auth:
    type: ldap
    host: ldap.example.com
    port: 389
    base_dn: dc=example,dc=com
    bind_dn: cn=admin,dc=example,dc=com
    bind_password: ${LDAP_BIND_PASSWORD}
    user_filter: "(uid={0})"

Observability

Prometheus Metrics

yaml
observability:
  prometheus:
    enabled: true
    port: 9090
    path: /metrics
bash
# Scrape metrics
curl http://localhost:9090/metrics

# Look for intu-specific metrics:
#   intu_messages_received_total
#   intu_messages_sent_total
#   intu_messages_errored_total
#   intu_pipeline_duration_ms

OpenTelemetry

yaml
observability:
  opentelemetry:
    enabled: true
    endpoint: http://localhost:4317
    service_name: intu
    insecure: true

Audit Logging

yaml
audit:
  enabled: true
  destination: memory    # memory or postgres
  events:
    - channel.deploy
    - channel.undeploy
    - channel.restart
    - message.reprocess

Audit events record:

  • User — who performed the action
  • Action — what was done (deploy, undeploy, restart, reprocess)
  • Target — channel ID or message ID
  • Timestamp — when the action occurred

Secrets Management

yaml
secrets:
  provider: env           # env | vault | aws_secrets_manager | gcp_secret_manager

# HashiCorp Vault example
secrets:
  provider: vault
  vault:
    address: https://vault.example.com
    token: ${VAULT_TOKEN}
    path: secret/data/intu
Tip For local development, the env provider reads secrets from environment variables and the .env file. Switch to Vault, AWS Secrets Manager, or GCP Secret Manager for production deployments.
